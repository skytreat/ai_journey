# 基金信息管理分析系统需求规格说明书

## 主要功能需求

### 0. 项目概述

**项目目标**: 构建一个集基金数据采集、存储、量化分析和推荐于一体的基金信息管理系统，帮助投资者筛选优质基金。

**目标用户**: 个人投资者、基金分析师、理财顾问

**核心功能**:

- 基金数据自动采集与更新（从 akshare 数据源）
- 多维度基金业绩分析
- 智能基金推荐
- 自定义数据查询（类 KQL 查询语言）(可放入下一版本支持)

**技术栈**：

| 层级   | 技术选型                           | 说明         |
| ---- | ------------------------------ | ---------- |
| 前端   | Vue 3 + Element Plus + ECharts | 用户界面展示     |
| 后端   | .NET 8 Web API                 | REST API服务 |
| 数据库  | SQLite 3.x                     | 数据持久化存储    |
| 数据源  | akshare（Python脚本）              | 基金数据获取     |
| 任务调度 | .NET BackgroundService         | 定时数据更新     |

**技术选型理由**：

1. **后端使用.NET Web API**：
   - 开发者已有.NET背景，无需学习新语言，开发效率高
   - 强类型系统，编译期错误检查，减少运行时问题
   - Visual Studio调试工具完善，开发体验好
   - Entity Framework Core提供类型安全的ORM
   - 性能优异，适合数据处理和分析计算
2. **前端使用Vue 3**：
   - 学习曲线平缓，模板语法类似Razor
   - Element Plus组件丰富，适合后台管理系统
   - 中文文档完善，社区活跃
   - 与.NET Web API通过RESTful接口通信，前后端分离
3. **数据采集使用Python脚本**：
   - akshare是Python库，用Python调用最方便
   - .NET通过Process或gRPC调用Python脚本获取数据
   - 数据采集脚本独立，不影响主服务稳定性

### 0.2 数据流图

**数据采集流程**：

```
akshare API → Python采集脚本 → 数据清洗 → SQLite数据库
                    ↓
              .NET BackendService（定时调度）
                    ↓
              日志记录/异常处理
```

**数据查询流程**：

```
用户请求 → Vue 3前端 → .NET Web API → Entity Framework → SQLite数据库
                                              ↓
                                        缓存（可选）
                                              ↓
                                        返回JSON → 前端渲染
```

**定时更新流程**：

```
.NET BackgroundService → 调用Python脚本 → 数据采集 → 数据校验
        ↓                                               ↓
   定时触发（增量15:30/全量周日03:00）              业绩指标计算
        ↓                                               ↓
   更新状态通知                                  SQLite数据库更新
```

### 1. 基金数据库，

包括以下几张表。选择的数据库产品需要免费，轻量，并且支持强大的查询分析功能，比如支持复杂的join查询，聚合函数，窗口函数等。

### 1.1  基金基本信息表（fund\_basic\_info）

**主键**: 代码\
**包含字段**:

- 代码：VARCHAR(20)，基金唯一标识代码
- 名称：VARCHAR(100)，基金全称
- 基金类型：VARCHAR(20)，标准化后的类型（股票型/债券型/混合型/货币型/指数型/QDII/FOF/其他）
- 基金子类型：VARCHAR(20)，标准化后的类型（偏股/灵活/等）
- 份额类型：VARCHAR(10)，A/B/C 等份额标识，A份额为主份额
- 主基金代码：VARCHAR(20)，非A份额基金对应的主基金代码，A份额此字段为空
- 成立日期：DATE，基金成立日期
- 上市日期：DATE，基金上市交易日期，未上市可为空
- 基金管理人：VARCHAR(100)，基金管理公司名称
- 基金托管人：VARCHAR(100)，基金托管银行名称
- 管理费率：VARCHAR(20)，年管理费率（%）
- 托管费率：VARCHAR(20)，年托管费率（%）
- 销售服务费率：VARCHAR(20)，销售服务费率（%），C类份额通常有此费用
- 业绩比较基准：VARCHAR(500)，基金业绩比较基准描述
- 跟踪标的：VARCHAR(100)，指数基金跟踪的标的指数名称
- 投资风格：VARCHAR(20)，价值型/成长型/平衡型等
- 风险等级：VARCHAR(10)，低风险/中低风险/中风险/中高风险/高风险
- 更新日期：DATETIME，数据更新时间

**说明**:

- 基金类型和基金子类型需要根据实际值的集合标准化，避免使用枚举值，因为枚举值会限制数据的灵活性和可扩展性。比如akshare获取的原始基金类型是，“混合型-偏股”，解析出来的基金类型是混合型，基金子类型是偏股。
- 完整基金类型列表：股票型、债券型、混合型、货币型、指数型、QDII、FOF、商品型、其他
- 完整基金子类型列表：偏股、灵活、等型
- 份额类型：A（前端收费）、B（后端收费）、C（无申购费但有销售服务费）等
- 对于非A份额基金，主基金代码为A份额基金的代码，其余字段为空

### 1.2 基金资产规模表（fund\_asset\_scale）

**主键**: (代码，日期)\
**包含字段**:

- 代码：VARCHAR(20)，基金代码
- 日期：DATE，规模统计日期
- 资产规模：DECIMAL(12,2)，单位：亿元
- 份额规模：DECIMAL(12,2)，单位：亿份
- 更新日期：DATETIME，数据更新时间

### 1.3 基金经理表（fund\_manager）

**主键**: (代码，基金经理姓名，任职开始日期)\
**包含字段**:

- 代码：VARCHAR(20)，基金代码
- 基金经理姓名：VARCHAR(50)，基金经理姓名
- 任职开始日期：DATE，开始管理该基金的日期
- 任职结束日期：DATE，结束管理该基金的日期，NULL表示仍在任
- 管理天数：INT，任职期间的管理天数
- 更新日期：DATETIME，数据更新时间

### 1.4 基金申购状态表（fund\_purchase\_status）

**主键**: (代码，日期)\
**包含字段**:

- 代码：VARCHAR(20)，基金代码
- 日期：DATE，状态记录日期
- 申购状态：VARCHAR(20)，枚举值：开放申购/限制申购/暂停申购/封闭期
- 申购限额：DECIMAL(12,2)，单日申购限额（元），NULL表示无限制
- 申购手续费率：DECIMAL(5,4)，标准申购费率（%）
- 更新日期：DATETIME，数据更新时间

### 1.5 基金赎回状态表（fund\_redemption\_status）

**主键**: (代码，日期)\
**包含字段**:

- 代码：VARCHAR(20)，基金代码
- 日期：DATE，状态记录日期
- 赎回状态：VARCHAR(20)，枚举值：开放赎回/限制赎回/暂停赎回/封闭期
- 赎回限额：DECIMAL(12,2)，单日赎回限额（元），NULL表示无限制
- 赎回手续费率：DECIMAL(5,4)，标准赎回费率（%）
- 更新日期：DATETIME，数据更新时间

### 1.6 基金历史净值表（fund\_nav\_history）

**主键**: (代码，日期)\
**包含字段**:

- 代码：VARCHAR(20)，基金代码
- 日期：DATE，净值日期
- 单位净值：DECIMAL(10,4)，单位净值（每日公布）
- 累计净值：DECIMAL(10,4)，累计单位净值（包含分红影响）
- 日增长率：DECIMAL(8,4)，日增长率（%），来自于akshare提供的日增长率，采用分红再投资算法计算：(当日净值 + 当日分红) / 前一日净值 - 1
- 更新日期：DATETIME，数据更新时间

**说明**:
- **只存原始数据**：单位净值和累计净值是基金公司公布的原始数据，不会随时间变化
- 分红、拆分等公司行为事件存储在独立的 fund\_corporate\_actions 表中

### 1.7 基金公司行为表（fund\_corporate\_actions）

**主键**: id（自增ID）\
**唯一索引**: (代码，除权日期，事件类型)\
**包含字段**:

- id：INTEGER，自增主键
- 代码：VARCHAR(20)，基金代码
- 除权日期：DATE，除权除息日
- 事件类型：VARCHAR(20)，枚举值：DIVIDEND(分红)/SPLIT(拆分)/MERGE(合并)/TRANSFORM(转型)
- 每份分红金额：DECIMAL(10,4)，每份分红金额（元），仅分红事件有效
- 分红发放日：DATE，分红实际发放到账户的日期
- 拆分比例：DECIMAL(10,4)，拆分比例（如1.5表示1份拆为1.5份），仅拆分事件有效
- 权益登记日：DATE，权益登记日，确定享有权益的持有人名单日期
- 事件描述：VARCHAR(500)，事件详细描述
- 公告日期：DATE，基金公司发布公告的日期
- 更新日期：DATETIME，数据更新时间

**说明**:

- 本表存储基金公司行为事件（公司行为/Corporate Actions），包括分红、拆分、合并、转型等
- 与 fund\_nav\_history 表通过(代码, 除权日期)关联
- 分红和拆分是稀疏数据（大部分基金每年0-4次），独立存储避免历史净值表大量NULL值
- 未来可扩展新的事件类型，无需修改表结构

**数据约束**:

- 每份分红金额必须大于0（分红事件）
- 拆分比例必须大于0（拆分事件）
- 同一基金同一天同一类型的事件只能有一条记录

### 1.8 基金业绩表（fund\_performance）

**主键**: id（自增ID）\
**唯一索引**: (代码，周期类型，周期值)\
**包含字段**:

- id：INTEGER，自增主键，用于关联查询和性能优化
- 代码：VARCHAR(20)，基金代码（只存A类份额代码）
- 周期类型：VARCHAR(20)，枚举值：成立以来/今年以来/近1周/近1月/近3月/近6月/近1年/近3年/历史年份
- 周期值：VARCHAR(20)，当周期类型为"历史年份"时，值为具体年份如"2023"；其他情况为空或与周期类型相同
- 净值增长率：DECIMAL(8,4)，百分比（%）
- 最大回撤：DECIMAL(8,4)，百分比（%）
- 下行标准差：DECIMAL(8,4)
- 夏普比率：DECIMAL(8,4)
- 索提诺比率：DECIMAL(8,4)
- 卡玛比率：DECIMAL(8,4)
- 年化收益率：DECIMAL(8,4)，百分比（%）
- 波动率：DECIMAL(8,4)，百分比（%）
- 同类型基金排名：INT
- 同类型基金总数：INT
- 更新日期：DATETIME，数据更新时间

**说明**:

- **只计算A类份额业绩**：非A类份额（B/C/E等）不计算业绩，通过主基金代码关联A类业绩
- **原因**：同一基金的不同份额投资标的相同，净值走势完全一致；避免重复计算和数据冗余
- **关联查询**：B/C类基金详情页通过 fund\_basic\_info.主基金代码 关联A类业绩数据
- 确保排名、多因子评估等分析的准确性和一致性

### 1.9 用户自选基金表（user\_favorite\_funds）

**主键**: id（自增ID）\
**唯一索引**: (用户ID, 基金代码)\
**包含字段**:

- id：INTEGER，自增主键
- 用户ID：VARCHAR(50)，用户唯一标识（V1版本可用设备ID或会话ID）
- 基金代码：VARCHAR(20)，基金代码
- 添加时间：DATETIME，加入自选的时间
- 排序序号：INT，用户自定义排序，默认按添加时间倒序
- 备注：VARCHAR(500)，用户自定义备注信息
- 分组标签：VARCHAR(50)，用户自定义分组（如"观察中"、"已持有"）
- 提醒设置：VARCHAR(200)，JSON格式，存储价格提醒、涨跌提醒等设置
- 更新日期：DATETIME，记录更新时间

**说明**:

- V1版本：单用户模式，用户ID可用固定值或设备标识
- V2版本：多用户模式，关联用户表
- 支持用户对自选基金进行分组管理和排序
- 支持设置价格提醒、涨跌幅提醒等个性化功能

### 1.10 自选基金多因子评分表（user\_favorite\_scores）

**主键**: id（自增ID）\
**唯一索引**: (用户ID, 基金代码, 评分日期)\
**包含字段**:

- id：INTEGER，自增主键
- 用户ID：VARCHAR(50)，用户唯一标识
- 基金代码：VARCHAR(20)，基金代码
- 评分日期：DATE，评分计算日期
- 综合评分：DECIMAL(5,2)，多因子综合得分（0-100分）
- 收益因子得分：DECIMAL(5,2)，收益类因子得分（0-100分）
- 风险因子得分：DECIMAL(5,2)，风险类因子得分（0-100分，越高越好）
- 风险调整收益得分：DECIMAL(5,2)，夏普等比率得分（0-100分）
- 排名因子得分：DECIMAL(5,2)，同类排名得分（0-100分）
- 评分排名：INT，在自选基金中的排名
- 评分变化：DECIMAL(5,2)，相比上期的评分变化
- 评分趋势：VARCHAR(10)，上升/下降/持平
- 权重配置：VARCHAR(500)，JSON格式，用户自定义权重配置
- 计算时间：DATETIME，评分计算时间
- 更新日期：DATETIME，记录更新时间

**说明**:

- 为用户的自选基金单独计算多因子评分
- 支持用户自定义权重配置，个性化评分模型
- 每日定时计算并缓存评分结果
- 支持查看评分历史趋势和变化
- 可用于自选基金内部排序和筛选

**与fund\_performance表的区别**:

- fund\_performance：全市场基金业绩，统一权重
- user\_favorite\_scores：仅自选基金，支持用户自定义权重

### 1.11 基金信息更新历史表（fund\_info\_update\_history）

**主键**: id（自增ID）\
**唯一索引**: (代码，更新日期，更新类型)\
**包含字段**:

- id：INTEGER，自增主键
- 基金代码列表：VARCHAR(5000)，JSON格式，存储更新的基金代码列表，全量更新时为空
- 更新类型：VARCHAR(20)，枚举值：FULL/INCREMENTAL
- 更新表名：VARCHAR(30)，更新的数据库表名，如：fund_basic_info、fund_nav_history等
- 更新状态：VARCHAR(20)，枚举值：SUCCESS(成功)/FAILED(失败)/PARTIAL(部分成功)/RUNNING(进行中)
- 更新记录数：INT，本次更新影响的记录数量
- 新增记录数：INT，本次更新新增的记录数量
- 更新耗时：INT，更新耗时（秒）
- 错误信息：VARCHAR(5000)，更新失败时的错误描述，成功时为空
- 更新来源：VARCHAR(50)，数据来源标识，如：akshare/manual/scheduled
- 操作人：VARCHAR(50)，执行更新的用户或系统标识
- 更新日期：DATETIME，记录更新时间

**说明**:

- 记录每次数据更新的详细情况，用于数据追溯和问题排查
- 支持按更新类型、时间范围查询更新历史
- 更新状态为FAILED时，错误信息字段记录详细错误原因
- 可用于监控数据更新频率和成功率
- 支持数据更新审计和合规要求


### 1.12 数据量预估

**数据规模估算**：

| 数据类型      | 数量级     | 单条大小   | 总大小         | 说明             |
| --------- | ------- | ------ | ----------- | -------------- |
| 基金基础信息    | 100,000条 | \~500B | 50MB         | 基金列表           |
| 历史净值数据    | 1,250万条 | \~50B  | 625MB       | 10,000只×1,250天 |
| 公司行为事件    | 5万条     | \~200B | 10MB        | 分红、拆分等         |
| 基金业绩数据    | 200万条    | \~50B | 100MB        | 多周期业绩          |
| 基金经理信息    | 2万条     | \~300B | 6MB         | 经理档案           |
| 资产规模历史    | 50万条    | \~30B  | 15MB        | 规模变动           |
| **数据库总计** | -       | -      | **\~750MB** | 含索引和冗余         |
| 日增量       | \~1万条   | -      | \~500KB     | 每日新增净值         |
| 年增量       | \~250万条 | -      | \~125MB     | 全年新增数据         |

**存储规划建议**：

- **V1版本**：SQLite单文件，预留2GB空间
- **数据增长**：预计3年后数据量约1.5GB
- **备份策略**：每日备份，保留15天，约需20GB存储
- **性能考虑**：当数据量超过2GB时，建议评估PostgreSQL

### 1.13 数据库选型

- **选型**: SQLite 3.x

#### 2.1.1 数据更新并发控制

V1版本使用简单的文件锁机制防止并发更新，更新期间查询返回"数据更新中"提示。

> **V2版本计划**（可选）：
>
> 1. **并发控制架构**
>    ```
>    ┌─────────────────────────────────────────────────────────┐
>    │                    数据更新状态机                        │
>    ├─────────────────────────────────────────────────────────┤
>    │  IDLE(空闲) → UPDATING(更新中) → VALIDATING(校验中)    │
>    │       ↑                            ↓                    │
>    │       └──────── ACTIVE(可用) ←─────┘                    │
>    └─────────────────────────────────────────────────────────┘
>    ```
> 2. **状态机设计**
>    - IDLE(空闲) → UPDATING(更新中) → VALIDATING(校验中) → ACTIVE(可用)
>    - 状态说明：
>      | 状态         | 说明        | 查询服务       | 更新服务  |
>      | ---------- | --------- | ---------- | ----- |
>      | IDLE       | 空闲状态，等待更新 | ✅ 正常       | ✅ 可启动 |
>      | UPDATING   | 数据更新中     | ⚠️ 返回"更新中" | ✅ 进行中 |
>      | VALIDATING | 数据校验中     | ⚠️ 返回"更新中" | ✅ 校验中 |
>      | ACTIVE     | 更新完成，数据可用 | ✅ 正常       | ❌ 需等待 |
> 3. **并发控制机制**
>    1. **更新锁（Update Lock）**
>       - 使用数据库乐观锁或文件锁实现
>       - 防止多个更新任务同时执行
>       - 锁超时时间：30分钟（防止死锁）
> 4. **读写分离策略**
>    ```
>    主数据库（写入）    副本数据库（读取）
>         ↓                    ↓
>    [数据更新] ←────── [快照同步]
>         ↓                    ↓
>    [事务提交] ←────── [切换读取]
>    ```
>    - 更新时：写入主数据库，查询走副本
>    - 更新完成：原子切换，查询指向新数据
> 5. **蓝绿部署方式**
>    ```
>    步骤1：创建临时表（蓝色环境）
>           fund_nav_history_temp
>
>    步骤2：写入新数据到临时表
>           不影响生产表查询
>
>    步骤3：数据校验
>           对比记录数、关键指标
>
>    步骤4：原子切换（事务内完成）
>           RENAME fund_nav_history → fund_nav_history_old
>           RENAME fund_nav_history_temp → fund_nav_history
>           DROP fund_nav_history_old
>    ```

> 6. **事务保障**
>    - 单表更新使用数据库事务
>    - 跨表更新使用分布式事务或补偿机制
>    - 事务隔离级别：READ COMMITTED
> 7. **数据一致性保障**
>    | 场景   | 策略   | 实现方式        |
>    | ---- | ---- | ----------- |
>    | 更新中断 | 回滚机制 | 事务回滚，保留原数据  |
>    | 更新失败 | 重试机制 | 最多重试3次，指数退避 |
>    | 数据损坏 | 备份恢复 | 从事务前快照恢复    |
>    | 并发查询 | 快照隔离 | 查询基于更新前快照   |
> 8. **更新流程时序**
>    ```
>    T1: 获取更新锁（失败则返回"更新中"）
>    T2: 创建临时表/快照
>    T3: 执行数据更新（写入临时表）
>    T4: 数据校验（记录数、完整性、准确性）
>    T5: 原子切换（事务内完成）
>    T6: 释放更新锁
>    T7: 清理临时数据
>    T8: 发送更新完成通知
>    ```

public class FundDataUpdateService : BackgroundService

#### 2.6 复权净值计算方法（直接使用akshare提供的分红再投资净值日增长率）

**说明**：akshare不直接提供复权净值，需要通过单位净值和累计净值计算获得。

**计算原理**：
复权净值反映的是将分红再投资后的净值走势，用于准确计算长期收益率。累计净值已经包含了分红的影响，精确的说，这里是后复权净值，可以通过以下公式计算复权净值：

```
复权净值 = 当日累计净值 / 首日累计净值 × 首日单位净值
```

**计算步骤**：

1. 调用 `ak.fund_open_fund_info_em(symbol=code, indicator="单位净值走势")` 获取单位净值
2. 调用 `ak.fund_open_fund_info_em(symbol=code, indicator="累计净值走势")` 获取累计净值
3. 合并两个数据集（按日期关联）
4. 按上述公式计算复权净值
5. 计算复权净值日增长率：`pct_change(复权净值) × 100`

**参考实现**：

```python
import akshare as ak
import pandas as pd

def calculate_adjusted_nav(symbol: str) -> pd.DataFrame:
    """计算基金复权净值"""
    # 获取单位净值和累计净值
    nav_df = ak.fund_open_fund_info_em(symbol=symbol, indicator="单位净值走势")
    accum_df = ak.fund_open_fund_info_em(symbol=symbol, indicator="累计净值走势")
    
    # 合并数据
    df = pd.merge(nav_df, accum_df, on="净值日期")
    df.columns = ["date", "nav", "daily_return", "accumulated_nav"]
    
    # 计算复权净值
    first_accum = df["accumulated_nav"].iloc[0]
    first_nav = df["nav"].iloc[0]
    df["adjusted_nav"] = df["accumulated_nav"] / first_accum * first_nav
    
    # 计算复权净值增长率
    df["adjusted_daily_return"] = df["adjusted_nav"].pct_change() * 100
    
    return df
```

**复权净值的动态特性**：

复权净值具有**动态变化**的特点：当基金发生新的分红时，**历史日期的复权净值会随之改变**。

**示例说明**：
假设某基金在2024-06-02分红0.05元：

- 站在2024-06-01看2024-03-01的复权净值：1.0800
- 站在2024-12-01看2024-03-01的复权净值：1.2800（已包含分红影响）

**原因**：复权净值计算公式 `复权净值 = 当前累计净值 / 首日累计净值 × 首日单位净值` 中，累计净值会随着新分红而增加。

**向前复权 vs 向后复权**：

| 复权方式     | 特点               | 本系统选择 |
| -------- | ---------------- | ----- |
| **向前复权** | 以最新日期为基准，历史数据会变化 | ✅ 采用  |
| **向后复权** | 以首日为基础，最新数据会变化   | 不采用   |

**本系统采用向前复权策略**，原因：

1. 最新净值保持不变，符合直觉
2. 历史收益反映"如果今天投资，过去表现如何"
3. 业界标准做法

**数据存储策略**：

- **fund\_nav\_history表**：只存单位净值和累计净值（原始数据）
- **复权净值**：查询时实时计算，不存储快照
- **fund\_performance表**：业绩数据在每次数据更新时重新计算

**注意事项**：

- 累计净值法计算简单快速，满足大多数分析场景
- 如需更精确的计算，可使用分红送配详情逐笔调整（复杂度较高，V2版本考虑实现）
- 新基金或数据不完整的基金可能无法计算复权净值
- 历史业绩数据会随新分红而更新，这是向前复权的正常特性

**输入参数：**

- 基金类型：string（如：混合型）
- 周期：enum（周/月/季/年）
- N：int（Top-N 和 Bottom-N 的数量）

**计算逻辑：**

1. 确定当前周期（如：本周）
2. 确定上周期（如：上周）
3. 计算两种变化指标：

   **A. 绝对变化值（推荐）**：
   - 绝对变化 = 当前周期净值增长率 - 上周期净值增长率
   - 适用场景：直观反映收益率的变化幅度
   **B. 相对变化率（辅助参考）**：
   - 当上周期净值增长率 > 0 时：变化率 = （当前周期净值增长率 - 上周期净值增长率）/ 上周期净值增长率
   - 当上周期净值增长率 = 0 时：变化率 = 当前周期净值增长率（或标记为极大正值）
   - 当上周期净值增长率 < 0 时：变化率 = 当前周期净值增长率 - 上周期净值增长率（不使用除法，避免误导）
4. 按绝对变化值排序筛选 Top-N/Bottom-N

**说明**：

- 默认使用**绝对变化值**进行排名，避免负增长情况下的误导
- 相对变化率仅作为参考指标展示，不参与排名
- 示例：上周-5%，本周-3%，绝对变化为+2%（改善），相对变化为-40%（易误解）

**输出：**

- Top-N 相比上周期净值增长率基金列表
- Bottom-N 相比上周期净值增长率基金列表

#### 3.3 多周期一致性筛选

**输入参数：**

- 基金类型：string
- 时间范围：\[start\_date, end\_date]
- 分析间隔：enum（月/季/年）
- N：int
- X%：float（一致性阈值，如 50% 表示至少在 50% 的周期内进入 Top-N）

**逻辑说明：**

1. 将时间范围按间隔划分为 K 个周期
2. 每个周期筛选 Top-N 基金
3. 统计每只基金出现在 Top-N 的次数
4. 筛选出现次数 >= CEIL(K \* X%) 的基金（向上取整）

#### 3.4 多因子量化评估

**输入参数：**

- 基金类型：string
- 时间范围：\[start\_date, end\_date]
- T：int（返回 Top-T 基金）

**评估因子：**（需明确）

- 收益因子：年化收益率、累计收益率
- 风险因子：最大回撤、波动率、下行标准差
- 风险调整收益因子：夏普比率、索提诺比率、卡玛比率
- 排名因子：同类排名百分位
  **多因子评分模型**：
  采用加权评分法，各因子权重可配置，默认权重如下：

| 因子类别   | 具体因子    | 权重  | 说明           |
| ------ | ------- | --- | ------------ |
| 收益因子   | 年化收益率   | 25% | 收益能力核心指标     |
| 收益因子   | 累计收益率   | 10% | 历史收益表现       |
| 风险因子   | 最大回撤    | 15% | 风险控制能力（越低越好） |
| 风险因子   | 波动率     | 10% | 收益稳定性（越低越好）  |
| 风险因子   | 下行标准差   | 5%  | 下行风险（越低越好）   |
| 风险调整收益 | 夏普比率    | 15% | 单位风险收益       |
| 风险调整收益 | 索提诺比率   | 5%  | 下行风险调整收益     |
| 风险调整收益 | 卡玛比率    | 5%  | 回撤调整收益       |
| 排名因子   | 同类排名百分位 | 10% | 相对表现（越高越好）   |

**权重设计依据**：

1. **收益因子合计35%**：收益能力是基金评价的核心，年化收益率占25%体现对长期收益的重视
2. **风险因子合计30%**：风险控制与收益同等重要，最大回撤15%反映对本金保护的重视
3. **风险调整收益合计25%**：衡量风险收益性价比，夏普比率15%作为最常用风险调整指标
4. **排名因子10%**：同类排名反映相对竞争力，避免单一市场环境影响

**权重自定义**：

- 支持用户调整权重并保存为自定义方案
- 提供预设方案：保守型（风险权重+10%）、进取型（收益权重+10%）、平衡型（默认）

**评分计算**：

1. 对每个因子进行标准化处理（Z-score或Min-Max归一化）
2. 风险类指标（最大回撤、波动率等）取反，使其方向与收益一致
3. 加权求和得到综合评分：Score = Σ(因子值 × 权重)
4. 按综合评分排序，返回Top-T基金

**评分计算频率**：每日定时计算并缓存（非实时计算）

**参考实现**：具体计算逻辑可参考 fund\_recommendation\_multiple\_factors.py 文件

export const useFundStore = defineStore('fund', {
  state: () => ({

      fundType: '',
      manager: '',
      minReturn: null
    },
    sortBy: 'annualReturn', // 排序字段
    sortDesc: true,         // 是否降序
    loading: false,         // 加载状态
    pagination: {           // 分页信息
      page: 1,
      pageSize: 20,
      total: 0
    }
  }),
  
  getters: {
    // 获取筛选后的基金列表
    filteredFunds: (state) => {
      return state.fundList.filter(fund => {
        if (state.filters.fundType && fund.fundType !== state.filters.fundType) return false;
        if (state.filters.minReturn && fund.annualReturn < state.filters.minReturn) return false;
        return true;
      });
    }
  },
  
  actions: {
    // 获取基金列表
    async fetchFundList() {
      this.loading = true;
      try {
        const response = await api.getFunds({
          ...this.filters,
          sortBy: this.sortBy,
          sortDesc: this.sortDesc,
          ...this.pagination
        });
        this.fundList = response.data;
        this.pagination.total = response.total;
      } finally {
        this.loading = false;
      }
    },
    
    // 更新筛选条件

```typescript
// storage.ts - 统一存储管理
class StorageManager {
  // 带过期时间的localStorage
  setWithExpiry(key: string, value: any, ttl: number) {
    const item = {
      value,
      expiry: Date.now() + ttl
    };
    localStorage.setItem(key, JSON.stringify(item));
  }
  
  // 获取并检查过期
  getWithExpiry(key: string) {
    const itemStr = localStorage.getItem(key);
    if (!itemStr) return null;
    
    const item = JSON.parse(itemStr);
    if (Date.now() > item.expiry) {
      localStorage.removeItem(key);
      return null;
    }
    return item.value;
  }
}
```

### 5. 自定义查询模块

#### 5.1 功能说明

提供类Kusto Query Language（KQL）的查询语言，支持用户进行 ad-hoc 数据分析。

**实现范围**：

- **V1版本（当前）**：实现简化版KQL，仅支持最核心的3个操作符：`where`、`project`、`limit`
- **V2版本（后续迭代）**：增加`top`、`summarize`、`sort`等高级操作符
- **V3版本（可选）**：支持复杂嵌套查询和自定义函数

**V1版本限制说明**：
考虑到开发成本和优先级，V1版本仅实现基础查询功能，满足80%的常用查询场景。复杂统计分析建议通过专门的分析接口或导出数据后使用专业工具处理。

#### 5.2 支持语法

**基本语法结构**：

```kql
表名
| 操作符1
| 操作符2
| 操作符3
```

**V1版本支持的操作符**：

| 操作符     | 语法                      | 说明     | 示例                      |
| ------- | ----------------------- | ------ | ----------------------- |
| where   | `where 条件`              | 筛选条件   | `where 基金类型 == "混合型"`   |
| project | `project 字段1, 字段2, ...` | 选择字段   | `project 代码, 名称, 年化收益率` |
| limit   | `limit N`               | 限制返回条数 | `limit 100`             |

**支持的比较操作符**：`==`, `!=`, `>`, `<`, `>=`, `<=`, `and`, `or`

#### 5.3 KQL查询安全限制

**查询限制**：

- **最大返回条数**：1000条（防止大数据量查询导致内存溢出）
- **最大查询时间**：5秒（超时自动终止）
- **禁止的操作**：DELETE、UPDATE、DROP、INSERT等数据修改操作
- **允许的表白名单**：
  - `fund_basic_info` - 基金基础信息
  - `fund_nav_history` - 基金历史净值
  - `fund_performance` - 基金业绩数据
  - `fund_manager` - 基金经理信息
  - `fund_corporate_actions` - 公司行为事件

**防护措施**：

- **语句预解析**：拦截包含危险关键字的查询
- **参数化查询**：使用参数化方式执行，防止SQL注入
- **查询审计**：记录所有KQL查询日志，包含用户、时间、查询内容
- **频率限制**：单用户查询频率限制60次/分钟
- **权限控制**：只读权限，禁止任何数据修改操作

**错误处理**：

- 语法错误：返回详细的语法错误位置和说明
- 超时错误：提示"查询超时，请简化查询条件"
- 权限错误：提示"无权访问该表或执行该操作"

#### 5.4 语法示例

```kql
// 查询混合型基金业绩数据
fund_performance
| where 基金类型 == "混合型" and 周期类型 == "近1年"
| project 代码, 名称, 年化收益率

// 查询某基金的历史净值
fund_nav_history
| where 代码 == "000001"

| project 日期, 单位净值, 日增长率
| limit 30
```

**V2版本计划支持**：

- `top`：取前N条排序
- `sort`：排序操作
- `summarize`：聚合统计（需配合聚合函数）

#### 5.5 安全控制补充

- **查询白名单**：仅允许查询以下表：
  - fund\_basic\_info
  - fund\_nav\_history
  - fund\_performance
  - fund\_manager
  - fund\_asset\_scale
- **超时限制**：查询执行时间不超过 30 秒
- **结果限制**：单次返回不超过 1000 条（V1版本限制）
- **禁止操作**：禁止DDL操作（CREATE/ALTER/DROP），禁止跨库查询

#### 6.9 缓存策略

**缓存架构**：

```
┌─────────────────────────────────────────────────────────┐
│                      多级缓存架构                        │
├──────────────┬──────────────┬───────────────────────────┤
│   内存缓存    │   本地缓存    │      应用层缓存            │
│  (In-Memory) │  (SQLite)    │   (Response Cache)        │
├──────────────┼──────────────┼───────────────────────────┤
│ • 热点数据    │ • 历史净值    │ • API响应缓存             │
│ • 实时计算结果│ • 业绩数据    │ • 页面数据缓存            │
│ • 会话数据    │ • 排名结果    │ • 静态资源缓存            │
└──────────────┴──────────────┴───────────────────────────┘
```

**缓存内容与时效**：

| 缓存内容    | 存储位置  | TTL  | 更新策略    |
| ------- | ----- | ---- | ------- |
| 基金基本信息  | 内存缓存  | 1小时  | 数据更新时失效 |
| 基金列表    | 内存缓存  | 30分钟 | 定时刷新    |
| 历史净值数据  | 本地缓存  | 24小时 | 增量更新    |
| 业绩排名数据  | 内存缓存  | 4小时  | 数据更新时失效 |
| 多因子评分结果 | 本地缓存  | 24小时 | 定时重算后更新 |
| 基金详情页数据 | 应用层缓存 | 10分钟 | 数据更新时失效 |
| KQL查询结果 | 内存缓存  | 5分钟  | 数据更新时失效 |

**缓存更新机制**：

- **主动失效**：数据更新时，主动清除相关缓存
- **定时刷新**：对热点数据设置定时刷新任务
- **懒加载**：缓存 miss 时从数据库加载并写入缓存
- **预热机制**：系统启动时预加载常用数据

**缓存一致性保障**：

- 数据更新采用事务，确保数据库与缓存一致性
- 使用缓存版本号，避免脏读
- 关键数据设置短TTL，平衡性能与一致性

**监控指标体系**：
